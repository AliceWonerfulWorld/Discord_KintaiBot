# 技術スタック

---

## 0️⃣ 前提定義（最初に決める）

| 項目        | 内容                                 |
| --------- | ---------------------------------- |
| プロダクト種別   | 社内・コミュニティ向け業務効率化ツール（Discord Bot + Web管理画面） |
| 想定ユーザー規模  | 1サーバーあたり数十〜数百人（初期は個人〜小規模チーム） |
| 可用性目標     | ベストエフォート（無料枠のホスティングサービスに依存するため） |
| セキュリティ要件  | Discord ID等のユーザー情報の適切な管理、他人のデータを書き換えられない権限管理 |
| パフォーマンス要件 | 打刻コマンドに対して3秒以内にレスポンスを返す |
| チーム体制     | 個人開発（1名） |
| リリース頻度    | 随時（アジャイル・機能追加ごとにデプロイ） |
| 予算制約      | 初期・運用ともに **0円（完全無料枠の組み合わせ）** |
| 既存資産      | なし |

# 1️⃣ 技術スタック構成

---

## 🖥 フロントエンド（Web管理画面）

| 項目      | 採用技術 | 採用理由 | 評価観点                     | 備考 |
| ------- | ---- | ---- | ------------------------ | -- |
| 言語      | TypeScript | Bot側と共通化可能、型によるバグ防止 | 型安全性 / 学習コスト / 保守性       | |
| フレームワーク | Next.js (App Router) | Reactエコシステム、Vercelへのデプロイが容易 | エコシステム / SSR可否 / パフォーマンス | |
| UIライブラリ | Tailwind CSS + shadcn/ui | 開発速度が早く、モダンでカスタマイズ性の高いUIが作れる | デザイン一貫性 / 開発速度           | |
| 状態管理    | React Context / SWR | 小中規模のためRedux等は過剰、非同期データフェッチはSWR等で完結 | スケール耐性 / Server State分離  | |
| フォーム管理  | React Hook Form + Zod | 打刻修正フォームなどのバリデーションと型安全を担保 | パフォーマンス / バリデーション        | |
| テスト     | (初期フェーズは割愛) | 開発スピードを優先するため、MVP以降で導入検討 | カバレッジ / 実行速度             | |

---

## 🧠 バックエンド（Discord Bot & API）

| 項目           | 採用技術 | 採用理由 | 評価観点             | 備考 |
| ------------ | ---- | ---- | ---------------- | -- |
| 言語           | TypeScript | フルスタックで言語を統一し、コンテキストスイッチを減らす | 生産性 / 型安全 / 採用市場 | Node.js環境 |
| 実行環境         | Node.js | エコシステムが巨大で、Discord Bot関連のドキュメントが豊富 | 成熟度 / エコシステム     | |
| フレームワーク      | discord.js | Discord APIのラッパーとしてデファクトスタンダード | 構造化 / 拡張性        | Bot本体に使用 |
| API方式        | REST API (Supabase Client) | DBへの直接アクセスで工数削減。複雑な処理はNext.js API Routes | 型安全 / 柔軟性 / 過不足  | |
| ORM / DBアクセス | Supabase SDK (supabase-js) | SupabaseのPostgreSQLとシームレスに連携でき、型生成も容易 | 型統合 / マイグレーション管理 | |
| 認証           | Supabase Auth | DiscordのOAuthログインが標準で簡単に組み込めるため | セキュリティ / OAuth対応 | |
| 非同期処理        | Event Listenerベース | Discordからのコマンド(Interaction)を受け取るための標準仕様 | 再試行 / 耐障害性       | |

## 🗄 データベース

| 項目    | 採用技術 | 採用理由 | 評価観点          | 備考 |
| ----- | ---- | ---- | ------------- | -- |
| メインDB | Supabase (PostgreSQL) | 無料枠が広く、BaaSとしてAPI・認証機能も揃っているため | ACID / スケール性  | |
| キャッシュ | なし | MVPフェーズではDBの直接参照で十分なパフォーマンスが出るため | レイテンシ改善       | |
| 検索    | PostgreSQL標準機能 | 複雑な全文検索は想定しないため標準のクエリで対応 | 全文検索性能        | |
| 分析基盤  | Supabase Dashboard | 標準のダッシュボード上で直接データ確認や簡単な集計が可能 | BI連携 / ETL容易性 | |

## ☁ インフラ / DevOps

| 項目     | 採用技術 | 採用理由 | 評価観点          | 備考 |
| ------ | ---- | ---- | ------------- | -- |
| ホスティング | [Web] Vercel<br>[Bot] Render または Fly.io | どちらも無料枠があり、GitHub連携でデプロイが簡単なため | 可用性 / コスト     | Botは常時稼働が必要 |
| コンテナ   | Docker | ローカル環境の統一、および将来的なインフラ移行の準備として | 再現性 / 可搬性     | ローカル用 |
| IaC    | なし | MVPフェーズではGUI/CLIでの手動構築で十分なため | 構成管理 / 再現性    | |
| CI/CD  | GitHub Actions | Linter/Formatterの実行、メインブランチマージ時の自動デプロイ | 自動化 / 安定性     | |
| 監視     | Vercel Analytics | フロントエンドの簡易的なパフォーマンス監視 | 可観測性 / アラート精度 | |
| ログ管理   | Render / Vercel 標準ログ | 各ホスティングサービスのコンソールで出力を確認 | トレーサビリティ      | |
| CDN    | Vercel Edge Network | Next.jsのデプロイに伴い標準で利用可能 | レイテンシ最適化      | |

## 🔐 セキュリティ

| 項目    | 方針 | 評価観点          |
| ----- | -- | ------------- |
| 認証方式  | Supabase Auth による Discord OAuthログイン | 標準準拠 / 拡張性    |
| 認可    | Supabase RLS (Row Level Security) で自身のデータのみ編集可能にする | RBAC / ABAC可否 |
| 通信    | ホスティングプロバイダ(Vercel等)によるTLS(HTTPS)の強制 | TLS強制 / 証明書管理 |
| データ保護 | 環境変数( `.env` )による各種トークンの厳重管理、リポジトリへのコミット防止 | 暗号化 / マスキング   |
| 脆弱性対策 | GitHub Dependabot による依存関係の自動アップデート検知 | 自動検査 / パッチ管理  |
